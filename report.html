This uniform interface lets main.py and app.py treat all algorithms identically, making it trivial to add a new method without touching the rest of the code.

4. Algorithm Implementations

4a. Non-Personalised Baselines baselines.py

Three baselines that make no use of the individual user's history. They serve as sanity checks: any personalised method should beat Random, and should aspire to beat MostPopular on accuracy.

MostPopular

Recommends the movies with the highest number of ratings (a proxy for popularity). One global ranking, filtered per-user to exclude already-seen items.

counts = ratings.groupby("movieId").size()
ranking = counts.sort_values(ascending=False).index.tolist()

HighestRated

Recommends movies with the highest average rating, subject to a minimum of 20 ratings to prevent obscure movies with a single 5-star rating from dominating.

Random

Uniformly samples unseen movies at random. Lower bound for accuracy; naturally achieves high novelty and catalogue coverage.

4b. Collaborative Filtering collaborative.py

Both methods build a user × item rating matrix (missing = 0) and use cosine similarity to find similar items or users. Scoring is fully vectorised via NumPy matrix multiplication.

Item-Item CF (k = 20, min_support = 5)

Finds the k items most similar to each candidate item, then aggregates the target user's ratings of those neighbours.

score(u, i) = Σⱼ sim(i,j)·r(u,j) / Σⱼ |sim(i,j)|   (j = top-k rated neighbours)

The min_support=5 parameter zeros out all similarities involving items with fewer than 5 training raters. Without this, single-rater items have cosine similarity = 1.0 with every other item the same user rated (identical direction in user-space), producing artificially inflated scores of 5.0 that flood the top-10. With the filter, P@10 recovered from 0.000 to 0.004.

Known limitation: With 98.3% sparsity, only 35.8% of items have 5 or more raters. For the remaining 64.2%, the similarity column is zeroed out, so those items cannot be scored and rank at zero. This means Item-Item CF effectively operates on a restricted sub-catalogue, which limits recall. The low P@10 (0.004) is a direct consequence of dataset sparsity — not a bug. Loosening to min_support=2 was tested and made results worse (P@10 = 0.001), confirming that sparser items produce even noisier similarities.

User-User CF (k = 20)

Finds the k users most similar to the target user, then uses their ratings to predict scores for unseen items.

score(u, i) = Σₙ sim(u,n)·r(n,i) / Σₙ |sim(u,n)|   (n = top-k similar users)

4c. Content-Based Filtering content_based.py

Builds item vectors from movie metadata (genres + user tags) using TF-IDF, then builds a user profile from their rating history.

Item vectors

Each movie's text is the concatenation of its pipe-separated genres and all user-generated tags (lowercased, deduplicated). TF-IDF with max_features=5000 converts this to a sparse vector.

# Example: "Action Sci-Fi" + "mind-bending dystopia time-travel"
TfidfVectorizer(min_df=1, max_features=5000).fit_transform(item_texts)

User profile

A weighted sum of item vectors, weighted by the user's centred rating (rating minus their personal mean). Items rated above average pull the profile toward their genres; items rated below average push away.

profile(u) = Σᵢ (r(u,i) − mean_r(u)) · vector(i)

Scoring

Cosine similarity between the user profile and all unseen item vectors. Top-n items are returned.

Why genres + tags? Tags like "mind-bending" or "based on true story" carry semantic meaning that genre labels ("Drama") cannot express. Adding tags expands the TF-IDF vocabulary with ~3,000 terms that differentiate within-genre taste.

4d. Matrix Factorization matrix_factorization.py

Decomposes the user-item rating matrix using Truncated SVD (scikit-learn's randomised SVD, requiring no extra dependencies).

R ≈ U · Σ · Vᵀ   (50 latent factors)

The full reconstructed matrix R̂ = U·Σ·Vᵀ is stored at fit time. At inference, a single row lookup and argsort returns top-n recommendations in microseconds.

TruncatedSVD vs. proper Matrix Factorization: This implementation differs from full MF (ALS, BPR, SGD-MF) in three ways: (1) Missing ratings are filled with 0 — conflating "not seen" with "rated 0 stars". (2) No user/item bias terms are modelled, so predicted scores are not interpretable as ratings. (3) No L2 regularisation, which can overfit dense regions. A production system would use implicit ALS or SGD-MF to address all three. TruncatedSVD is used here for reproducibility without additional dependencies.

5. Evaluation Methodology

evaluation.py · data_loader.py

Train / Test Split — Temporal (Per-User)

For each of the 610 users, ratings are sorted chronologically and the most recent 20% are held out as the test set. The earlier 80% form the training set. This mirrors real deployment: models learn from past interactions and are evaluated on future ones, eliminating the temporal leakage that a random shuffle would introduce.

for _, group in ratings.groupby("userId"):
    group_sorted = group.sort_values("timestamp")
    n_test = max(1, int(len(group_sorted) * 0.2))
    train_parts.append(group_sorted.iloc[:-n_test])
    test_parts.append(group_sorted.iloc[-n_test:])

Every user is guaranteed at least one training rating. The resulting split: Train 80,896 · Test 19,940 (610 users evaluated).

Relevance Threshold

A test item is counted as "relevant" only if its rating is ≥ 3.5 stars. This follows the standard practice in RS literature (Herlocker et al., 2004): treating a 0.5-star rating as equally relevant as a 5-star one inflates recall and misrepresents recommendation quality. Users who rated no test items ≥ 3.5 are excluded from precision/recall computation (593 users remain eligible).

Key Observations

  1. Matrix Factorization dominates all accuracy metrics — P@10 = 0.090, NDCG@10 = 0.112, HR@10 = 0.498. The 50 latent dimensions capture collaborative taste patterns invisible to raw item-item similarity, and this advantage holds at K=5 and K=20.
  2. MostPopular is a strong baseline — P@10 = 0.064, second only to MatrixFact, with zero personalisation overhead. Its Coverage (1.0%) reveals the cost: it effectively recommends the same ~100 movies to everyone.
  3. The accuracy–novelty trade-off is stark — MatrixFact (highest accuracy) has Novelty=9.55 and AvgPop=126. ContentBased (highest novelty=14.64, highest coverage=32.1%) has P@10=0.008. No single model is best on both dimensions.
  4. ContentBased has the lowest ILD (0.201) — it clusters recommendations tightly by genre. This makes sense: the user profile is a weighted genre/tag vector, so all recommendations land in similar content territory. MostPopular has the highest ILD (0.952) because popular movies span many genres.
  5. ItemItemCF accuracy is limited by sparsity — only 35.8% of items have ≥5 co-raters. For the rest, similarities are zeroed out by the min_support=5 filter, so they cannot be ranked. This is a fundamental constraint of memory-based CF on 98.3%-sparse data, not a bug. The model's low AvgPop (7.3) shows it does surface genuinely niche items when it can form opinions.
  6. UserUserCF recall (0.017) exceeds its precision (0.010) — it finds a meaningful fraction of liked items, just not always in the top 10 slots. Its coverage (15.6%) is solid for a CF method.

7. Beyond-Accuracy Analysis

Popularity Bias

AvgPopularity (mean rating count of recommended items) reveals how blockbuster-heavy each model is. MostPopular averages 217 ratings per recommendation — almost exclusively top-tier films. MatrixFact (126) also leans popular, reflecting that latent factors trained on dense data are dominated by frequently-rated movies. ContentBased (9.6) and ItemItemCF (7.3) surface mostly long-tail films.

In practice: a system that recommends Forrest Gump and Pulp Fiction to every user adds no value for users who have already seen the classics. ContentBased's low popularity bias makes it a strong complement to MatrixFact in a hybrid system.

Novelty

Self-information novelty (Zhou et al., 2010) quantifies how "surprising" recommendations are. ContentBased (14.64) provides the most novel recommendations — its genre+tag profiles point to corner items of the catalogue. MostPopular (8.57) provides the least — it recommends items that essentially everyone has already heard of. This directly mirrors the popularity bias picture above.

Catalog Coverage

ContentBased recommends from 32.1% of the catalogue across all users — far more than any other method. This stems from the personalised genre+tag profile: each user's profile vector points to a different region of the item space. MostPopular's 1.0% coverage means it treats 9,742 movies as if only ~97 were worth recommending.

Intra-List Diversity (ILD)

ILD measures the average pairwise cosine dissimilarity within each top-10 list. The results reveal a striking pattern:

  • ContentBased: ILD = 0.201 — the lowest of all models. Recommendations cluster tightly by genre; a user who likes sci-fi gets 10 sci-fi recommendations with similar TF-IDF vectors.
  • MostPopular: ILD = 0.952 — the highest. Popular movies span many genres (action, drama, comedy, thriller), so the top-10 list is naturally diverse.
  • MatrixFact: ILD = 0.926 — high diversity despite high accuracy. Latent factors capture multi-genre taste patterns, producing varied lists.

This finding suggests ContentBased could benefit from a post-processing diversification step (e.g., MMR re-ranking) to avoid genre monotony while retaining content relevance.

The Accuracy–Diversity Trade-off

A key insight from recommender systems research: optimising for accuracy alone leads to filter bubbles and popularity bias. High-accuracy models (MatrixFact, MostPopular) converge on popular items and high ILD. High-novelty models (ContentBased) explore the long tail but sacrifice hit rate and intra-list variety. A production system would blend both — e.g., 80% MatrixFact + 20% ContentBased — to balance relevance with discovery, then apply MMR or DPP re-ranking for final list diversity.

8. User Interface — Streamlit Prototype

app.py

The Streamlit app provides a browser-based interactive interface. It is structured into three pages accessible from the sidebar:

Page 1: Dataset Insights

Five KPI cards (Users, Movies, Ratings, Sparsity, Avg Rating) followed by four charts: rating distribution histogram, ratings-per-user distribution, top-15 most-rated movies, and genre frequency. Page 2: Get Recommendations

The user selects any of the 610 users from a dropdown, picks an algorithm, and sets how many recommendations to show (5–20). The left panel shows the user's profile (top-rated movies, favourite genres). A cold-start warning appears if the user has fewer than 20 training ratings, recommending Content-Based or MostPopular as fallbacks. The right panel shows the ranked list with title, genres, popularity count, and — for Matrix Factorization — the predicted reconstruction score, making the model's confidence in each recommendation visible.

A styled metrics table (green = best, red = worst per column) with all 9 metrics for all 7 models. Five accuracy bar charts (P@10, R@10, NDCG@10, MRR@10, HR@10) are followed by four beyond-accuracy charts (Coverage, Novelty, AvgPopularity, ILD). A Key Takeaways section summarises the main findings. A download button exports results.csv.

9. Technical Challenges & Solutions

ChallengeSolution
Extreme sparsity (98.3%) Used TruncatedSVD (randomised, memory-efficient) rather than exact SVD. Stored the reconstructed matrix R̂ at fit time rather than recomputing at inference.
Vectorising CF scoring Both CF methods use fully vectorised NumPy operations (sim_matrix @ user_vec) instead of Python loops, reducing inference from O(n²) Python iterations to a single BLAS matrix multiplication.
ItemItemCF single-rater artefact Items rated by only one user have cosine similarity = 1.0 with every other item that user rated (identical direction in user-space). These flood the top-10 with scores equal to the user's highest rating. Fix: min_support=5 zeros similarities for items with fewer than 5 raters. P@10 recovered from 0.000 to 0.004. Lowering to min_support=2 was tested and worsened results (P@10 = 0.001), confirming that the issue is fundamental to dataset sparsity.
Temporal leakage in evaluation A random train/test shuffle leaks future ratings into training. Fixed with a per-user temporal split: each user's most recent 20% of ratings are held out, mimicking real deployment conditions.
Relevance definition Counting all test items as "relevant" regardless of rating inflates recall (a 0.5-star item treated the same as a 5-star one). Fixed by applying a relevance threshold of ≥3.5 stars, following Herlocker et al. (2004).
Content-based vocabulary richness Genres alone give only 19 features — too coarse to differentiate within-genre taste. Adding user-generated tags (lowercased, deduplicated per movie) expands the vocabulary to ~3,000 terms, capturing nuance like "visually stunning" or "based on true story".
Module portability Hard-coded relative paths break when the app is launched from a different working directory. Fixed using Path(__file__).parent to build absolute paths at import time.
Evaluation scope Original code evaluated only 50 of 610 users. Fixed by evaluating all eligible users (those appearing in both train and test splits with at least one liked test item).
Streamlit startup time Training 7 models including two cosine similarity matrices takes ~60 seconds. Used @st.cache_resource to train once per session; all subsequent interactions are instant.

10. Conclusions

Summary

This project implements a complete, modular recommender system covering all four major algorithm families. Every method is a runnable module with a consistent API. The evaluation framework goes beyond accuracy to measure diversity, novelty, intra-list variety, and popularity bias — capturing trade-offs that accuracy metrics alone conceal. A temporal train/test split and a relevance threshold of ≥3.5 stars ensure the evaluation reflects realistic deployment conditions rather than an optimistic random holdout.

Future Work

  • Hybrid model — blend MatrixFact (accuracy) with ContentBased (novelty) in a weighted ensemble; expected to push P@10 above 0.10 while maintaining coverage above 15%
  • Proper MF with bias terms — replace TruncatedSVD with SGD-MF or ALS (e.g., via surprise or implicit), adding per-user and per-item bias terms to improve predicted rating calibration
  • Implicit feedback — treat "viewed but not rated" as weak positive signal using BPR or ALS on binary interaction data
  • Parameter tuning — grid search over CF neighbourhood size k and SVD dimensionality n_factors to find optimal settings per metric
  • MMR re-ranking for ContentBased — post-process ContentBased lists with Maximal Marginal Relevance to raise ILD from 0.20 toward 0.50+ without sacrificing relevance
